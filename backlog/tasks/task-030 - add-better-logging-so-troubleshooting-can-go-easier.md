## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
Part 1/2 of the logging-troubleshooting plan (shared label: logging-troubleshooting). Goal: make setup-specific bugs from remote, non-technical users diagnosable by always writing a shareable Diagnostic log.

This task = CORE/backend logging infra only. GUI controls (toggle, open-folder, export-zip) are the dependent child task.

Decisions (from grill-with-docs 2026-06-17):
- Always-on rotating file at INFO. No user action to enable. Sink: platformdirs.user_log_dir('romhop'). RotatingFileHandler ~1MB x 5.
- Detailed logging (DEBUG) opt-in two ways: persisted setting 'debug_logging' (config SCHEMA FieldSpec, bool, default false) AND a global CLI -v/--verbose flag via a new Typer @app.callback. Either raises root logger INFO->DEBUG.
- Console: file always; stream to stderr ONLY when -v. Normal CLI typer-echo UX untouched. GUI is file-only.
- Redaction via a logging.Filter applied to all handlers (every level): API token/Authorization NEVER logged; RomM host masked to <romm-host>; user home dir abbreviated to ~ in paths.
- Add the log dir to config.purge_user_data() so uninstall wipes it.
- Format: '%(asctime)s %(levelname)s %(name)s: %(message)s'.
- Instrument key seams with INFO/DEBUG: settings load, RommClient requests (DEBUG, redacted), download start/finish/error, sync watch events, pull conflicts. Reuse existing module loggers (already present in gui/app, update, theme, covers, main_window).
- Keep backend Qt-free: logging setup lives in a core module (e.g. romhop/logging_setup.py), callable from both cli and gui.run().
<!-- SECTION:DESCRIPTION:END -->
