## Bug

On Windows (cmd.exe and PowerShell), `romhop --help` renders at ~80 cols even when the terminal is wider. The frog art takes ~25 cols; `body_w` is calculated as `max(24, console.width - 25 - 3 - 1)`. When `console.width` = 80, that gives body_w ≈ 51, so every description word-wraps into many lines → the output is very tall and doesn't use available width.

**Root cause:** `ru._get_rich_console().width` returns 80 on Windows for a PyInstaller-packaged .exe, even when the terminal is actually wider. `shutil.get_terminal_size()` queries the OS directly (`GetConsoleScreenBufferInfo` on Windows) and is more reliable.

**Screenshot confirmed:** wide terminal, body panel capped at ~30 chars, cmd.exe and PowerShell both affected, via .exe from releases.

**Relevant code:** `src/romhop/cli.py:103-199` — `_format_help_with_frog()`, specifically line 126:
```python
body_w = max(24, console.width - art_w - gap - left_pad)
```

## Agent Brief

> *This was generated by AI during triage.*

**Task:** Fix `romhop --help` terminal width detection on Windows

**File:** `src/romhop/cli.py`

**Change:** In `_format_help_with_frog()` at line 126, replace `console.width` with `shutil.get_terminal_size(fallback=(console.width, 24)).columns`:

```python
# before
body_w = max(24, console.width - art_w - gap - left_pad)

# after
import shutil  # (add at top of module or inside the function)
term_w = shutil.get_terminal_size(fallback=(console.width, 24)).columns
body_w = max(24, term_w - art_w - gap - left_pad)
```

`shutil` is stdlib — no new dependency. `fallback=(console.width, 24)` means if OS query fails, behavior is identical to today.

**Verify:** No automated way to test Windows terminal width on Linux CI. After the fix, do a manual smoke on Windows: run `romhop.exe --help` in a wide terminal (>100 cols) and confirm the panel fills the width.
