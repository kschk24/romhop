## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
`config.py` stores/reads the RomM API token via raw `keyring.set_password` / `keyring.get_password` (config.py:281-286) with no backend detection or fallback. On a frozen installer build running in an environment with no usable Secret Service backend (e.g. a minimal Linux session with no gnome-keyring/kwallet daemon), this fails silently or hard:

- `get_password` returns `None` → the app behaves as if no token is set (token appears "lost", user must re-enter every session).
- `set_password` can raise `keyring.errors.NoKeyringError` → potentially an unhandled traceback when saving the token.

This is the genuinely-actionable latent issue split out of TASK-038 (which was closed as not-reproducible). It is a different bug than TASK-038's reported symptom, but the same code area.

Goal: detect when the active keyring backend is missing / non-persistent (`fail.Keyring`, null/chainer-with-no-viable backend) and surface a clear, actionable error to the user instead of silently losing the token or crashing. Optionally fall back to an explicit, documented store. Decide fallback policy during planning — do not silently write the token to plaintext without a deliberate decision.
<!-- SECTION:DESCRIPTION:END -->

## Acceptance Criteria
<!-- AC:BEGIN -->

- [ ] #3 A documented fallback policy is implemented or explicitly decided against (no silent plaintext write without a deliberate decision).

## Ready-for-human brief

**Why human, not AFK agent:** the work is gated on a fallback-policy decision — if no persistent keyring backend exists, what does romhop do with the token? Options: (a) refuse + instruct the user to set up a secret-service/keyring, (b) write to an OS-permission-restricted encrypted file, (c) plaintext file with explicit warning. This is a security/product judgment call and must be made before the implementation can land. Once decided, the build itself is delegatable.

**Root area:** config.py:281-286 — `set_token`/`get_token` call raw `keyring.set_password`/`get_password` with no backend detection or fallback.

**Findings from triage:**
- Frozen build DOES bundle the backend modules (SecretService, libsecret, KWallet, Windows, macOS, chainer — see packaging/dist/.../keyring-25.7.0.dist-info/entry_points.txt). The runtime variable is whether a secret-service DAEMON is actually running. On a minimal Linux session with no gnome-keyring/kwallet daemon, the backend resolves but `get_password` returns None and `set_password` can raise `keyring.errors.NoKeyringError`.
- Therefore detection must probe an actual write→read round-trip of a sentinel value, NOT just inspect the resolved backend class name (a bundled class can resolve while no daemon answers).
- Existing tests already monkeypatch keyring (tests/test_config.py and 6 others), so faking an unavailable/non-persistent backend in the suite needs no real OS keyring.
- Empirically unconfirmed: no minimal-Linux repro was run. This hardens against a code-traced but unobserved failure (split out of TASK-038, which was closed as not-reproducible).

**Decision the human must make first:** the fallback policy (a/b/c above). Then implement detection + user-facing warning + chosen fallback per the acceptance criteria.
