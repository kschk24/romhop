## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
> *This was generated by AI during triage.*

DetailPanel image header flips between the RomM screenshot and the cover art when a game tile is clicked repeatedly. The image shown is nondeterministic per click.

**Root cause (confirmed in code):** `DetailPanel._start_cover_load` (src/romhop/gui/detail_panel.py:268-298) starts TWO async loaders for the same `_image_label` slot — a cover loader and a screenshot loader. Both connect to `_on_image_ready` → `_apply_pixmap`. The inline comment says "screenshot replaces cover once ready", but nothing enforces that ordering: whichever loader finishes LAST wins. Cover-vs-screenshot completion order is a race (cache hit vs network latency), so each click can land on either image.

**Fix direction:** make screenshot authoritative. Track per-rom which source is shown; once a screenshot has been applied for the current rom, ignore a later-arriving cover for that same rom. A cover may still fill the slot first (placeholder→cover) and be upgraded to a screenshot, but a screenshot must never be downgraded back to the cover. Tag each CoverLoader with its source so `_on_image_ready` can apply priority. Existing stale-loader cancellation on rom switch (lines 269-272) stays.
