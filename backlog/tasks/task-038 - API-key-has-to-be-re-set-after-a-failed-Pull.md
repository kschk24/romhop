The TASK-035 CLI repro showed NO token/auth problem — the 404 was a missing content blob, not a 401. So this symptom is not explained by that code path and is likely GUI-only (an error handler that drops the keyring token, forces re-auth, or invalidates the client on any pull error).

Needs investigation/repro before it can be specced:

- Is it tied to pull errors specifically, or any HTTP error in a worker?
- Trace the GUI pull/error path (PullWorker + main_window pull handler + token/keyring handling) to find what clears or appears to clear the token.

## Resolution: can't reproduce — superseded by TASK-035 fix

The symptom's trigger (a Pull aborting on an orphan-content HTTP 404) was fixed in TASK-035 (commits a67332f "skip orphan content 404s instead of aborting", 7bbfe35 "count orphan 404s as missing not failed"). With that path gone, the reporter can no longer reproduce the key-reset symptom.

Code trace (done 3x) found no mechanism that clears the keyring token on any pull/HTTP error: `config.set_token` is never called with a blank value, there is no `keyring.delete_password`, and `PullWorker`/`_on_pull_failed` only show a dialog. The reported "401 in same session" could not be re-verified.

Closing as not reproducible. A separate, genuinely-actionable latent risk (raw keyring with no backend detection on frozen builds) is split out into its own task.
